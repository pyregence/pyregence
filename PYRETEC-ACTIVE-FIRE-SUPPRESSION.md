# Suppress Pyretechnics Active-Fire Layers When Feature Is Disabled

## Jira Ticket

**Title:** Active fire forecasts display in prod despite pyretechnics being disabled

**Type:** Bug

**Priority:** Medium

**Labels:** frontend, geoserver, feature-flags

**Affects Version:** Production (pyrecast.com)

---

### Description

When the `:pyretechnics` feature flag is set to `false` in `config.edn` (as it is in production), active fire forecasts that were generated by the Pyretechnics model still appear in the UI. Users see fire names in the Active Fires tab dropdown but cannot view any forecast data for them because the only available model (ELMFIRE) has no corresponding layers on GeoServer.

### Steps to Reproduce

1. Set `:pyretechnics false` in `config.edn` (prod default).
2. Ensure GeoServer (`:trinity`) has fire-spread-forecast layers using the `pyretec` model but none using `elmfire`.
3. Navigate to the Active Fires tab.
4. Observe that fire names appear in the "Fire Name" dropdown.
5. Select any fire name — no forecast data loads because the default model is ELMFIRE and no ELMFIRE layers exist.

### Expected Behavior

When `:pyretechnics` is disabled, fires that only have Pyretechnics-model layers should not appear in the Active Fires tab. If no ELMFIRE active fires exist, `active-fire-count` should be 0, the default tab should fall back to Weather, and selecting the Active Fires tab should show the "no active fire forecasts" toast.

### Actual Behavior

Pyretechnics-only fires appear in the dropdown. Users can select them but see no data — a confusing dead-end experience.

---

## Root Cause Analysis

### Data Flow

1. **`capabilities.clj` / `process-layers!`** — Calls GeoServer's `GetCapabilities` and parses layer names via regex into the `layers` atom. Each layer name encodes the model (`elmfire` or `pyretec`) after the `:` separator:
   ```
   fire-spread-forecast_<fire-name>_<timestamp>:pyretec_landfire_<percentile>_<output>
   fire-spread-forecast_<fire-name>_<timestamp>:elmfire_landfire_<percentile>_<output>
   ```

2. **`capabilities.clj` / `get-fire-names`** — Reads the `layers` atom, collects distinct `:fire-name` values from layers with forecast `"fire-spread-forecast"`, and returns them as `{:active-fires {...} :match-drops {...}}`.

3. **`near_term_forecast.cljs` / `initialize!`** — Fetches fire names, sets `active-fire-count`, and populates `!/capabilities` with fire name options. If `active-fire-count > 0`, the Active Fires tab is the default.

4. **`config.cljs`** — The `:model` param under `:active-fire` has two options: `:elmfire` (always shown) and `:pyretechnics` (gated by `(feature-enabled? :pyretechnics)`). When the flag is off, only ELMFIRE is selectable.

### The Gap

`process-layers!` already had a feature-flag gate for **match-drop** layers:
```clojure
(or (get-config ... :features :match-drop) (not (str/includes? full-name "match-drop")))
```

But there was **no equivalent gate for pyretechnics** layers. So even with `:pyretechnics false`, pyretec layers were loaded into the atom, `get-fire-names` found them, and the frontend displayed fire names with no usable forecast data behind them.

---

## Fix

Added the same feature-flag pattern for pyretechnics in two `cond` branches inside `process-layers!`:

**File:** `src/clj/pyregence/capabilities.clj`

1. **Fire-spread-forecast branch (~line 198):**
   ```clojure
   (or (get-config :triangulum.views/client-keys :features :pyretechnics)
       (not (str/includes? full-name ":pyretec")))
   ```

2. **Isochrones branch (~line 203):**
   ```clojure
   (or (get-config :triangulum.views/client-keys :features :pyretechnics)
       (not (str/includes? full-name ":pyretec")))
   ```

When `:pyretechnics` is `false`, any layer whose name contains `:pyretec` is excluded at the `process-layers!` stage. Downstream:
- `get-fire-names` no longer returns pyretechnics-only fires.
- `active-fire-count` drops to 0 if no ELMFIRE fires exist.
- The default tab falls back to Weather.
- Fire-detection layers (VIIRS hotspots, GOES imagery, perimeters) are unaffected — they come from `:shasta`, not `:trinity`, and use a separate code path.

### Impact

- No changes to the frontend code.
- No changes when `:pyretechnics` is `true` (dev, staging) — all layers load as before.
- Fires that have both ELMFIRE and Pyretechnics layers will still appear (only the pyretec layers are filtered; ELMFIRE layers remain).
